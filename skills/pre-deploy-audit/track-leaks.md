# Трек A - утечки и экспозиция данных

Цель: найти всё, что **нельзя выкатывать или публиковать**. Метод - паттерны и пути, без понимания логики кода. Это «проверка на утечки».

Жёсткое правило (см. SKILL.md): не открывай secret-named файлы, не выводи значения. Улика - маска (первые/последние 2 символа), в отчёт идёт `file:line` + категория.

## Что ищем

### 1. Секреты и ключи в коде

Контентные паттерны по tracked-файлам:

| Категория | Паттерн (regex) |
|-----------|-----------------|
| Приватные ключи | `-----BEGIN (RSA\|EC\|OPENSSH\|DSA\|PGP)? ?PRIVATE KEY-----` |
| AWS access key | `AKIA[0-9A-Z]{16}` |
| AWS secret | `aws_secret_access_key\s*[:=]` + 40-символьная base64 |
| Google API | `AIza[0-9A-Za-z_\-]{35}` |
| Google OAuth | `[0-9]+-[0-9a-z]+\.apps\.googleusercontent\.com` |
| GitHub token | `gh[pousr]_[0-9A-Za-z]{36}` |
| Slack | `xox[baprs]-[0-9A-Za-z-]{10,}` |
| Stripe | `sk_live_[0-9A-Za-z]{24}`, `rk_live_` |
| JWT | `eyJ[A-Za-z0-9_-]+\.eyJ[A-Za-z0-9_-]+\.` |
| Generic | `(api[_-]?key\|secret\|token\|passwd\|password)\s*[:=]\s*['"][^'"]{6,}['"]` |
| Connection string | `(postgres\|mysql\|mongodb\|redis)://[^:]+:[^@]+@` (пароль в URL) |
| Private key files | имена `*.pem`, `*.key`, `*.p12`, `*.pfx`, `id_rsa`, `*.keystore` в индексе |

### 2. Закоммиченные секрет-файлы

`.env`, `.env.local`, `.env.production`, `.env.*`, `secrets.json`, `credentials.json`, `service-account*.json` - **в индексе git быть не должны**. Проверка: есть ли они в `git ls-files`. Если есть - Critical (для public).

### 3. .gitignore-ловушки

- Нужные паттерны (`.env*`, `*.pem`, `*.key`, секрет-файлы, артефакты сборки) **отсутствуют** в `.gitignore` - hardening (Low), но если рядом есть сами файлы - выше.
- **Ignored-but-tracked**: файл уже в индексе, а потом добавлен в `.gitignore`. `.gitignore` его не уберёт - он остаётся в репозитории и истории. Проверка: пересечение `git ls-files` с паттернами `.gitignore`.

### 4. PII и внутренняя инфраструктура (средний/полный)

- Email-адреса (`[a-z0-9._%+-]+@[a-z0-9.-]+\.[a-z]{2,}`), реальные имена в комментариях/данных.
- Внутренние IP (`10\.`, `192\.168\.`, `172\.(1[6-9]\|2[0-9]\|3[01])\.`), внутренние домены/хосты.
- Machine-paths: `C:\Users\<имя>`, `/home/<user>`, `/Users/<user>` - выдают пользователя и среду.
- Клиентские данные, дампы, реальные записи в фикстурах/seed.

### 5. История git (только полный, обязательно для public)

Секрет, удалённый из рабочего дерева, **остаётся в истории** и станет публичным вместе с репозиторием.

```bash
git log --all -p | grep -nEi '(-----BEGIN|AKIA[0-9A-Z]{16}|api[_-]?key|password|secret|token)'
# либо специализированно (если установлен): gitleaks detect / trufflehog git file://.
```

Любое попадание в истории при подготовке к public = Critical. Удалить из текущего коммита недостаточно.

### 6. Артефакты и бинарники

Закоммиченные `node_modules/`, `dist/`, `build/`, `.venv/`, крупные бинарники, дампы БД, логи. Раздувают репозиторий и иногда несут данные/пути.

## Уровни трека A

- **Поверхностный**: пп. 1-3 по рабочему дереву.
- **Средний**: + п. 4 (PII/инфра), п. 6.
- **Полный**: + п. 5 (история git), PII вглубь, supply chain.

## Сценарий «смена видимости на public»

Адаптация запроса «полный аудит перед тем как сделать репозиторий публичным». Чеклист трека A на полном уровне:

- [ ] Рабочее дерево: ни одного секрета/ключа/токена (пп. 1-2)
- [ ] **История git чиста** (п. 5) - проверена `git log -p` или gitleaks/trufflehog
- [ ] Нет ignored-but-tracked секрет-файлов (п. 3)
- [ ] `.gitignore` покрывает `.env*`, ключи, артефакты
- [ ] Нет внутренних хостов/IP, machine-paths, клиентских данных (п. 4)
- [ ] `LICENSE` соответствует намерению публикации (по умолчанию DotCore - строгий All Rights Reserved; публикация ≠ open source)
- [ ] README/доки не раскрывают приватную инфраструктуру

## Ремедиация (рекомендация, не автодействие)

При найденном живом секрете порядок строгий:

1. **Считай ключ скомпрометированным - ротация в первую очередь.** Удаление из git не отзывает ключ; если он был в публичной/общей истории, его уже могли забрать.
2. Убери из рабочего дерева, добавь в `.gitignore`, передавай через env/secret manager.
3. Если секрет в истории и репозиторий идёт в public - перепиши историю (`git filter-repo` или BFG) **вручную, по решению пользователя**. Скилл это не делает сам (деструктивно).

Не выполняй ротацию, удаление файлов и переписывание истории автоматически - только сообщи, что и в каком порядке нужно сделать.
