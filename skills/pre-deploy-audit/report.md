# Вердикт: severity, готовность, гейт

Финальный шаг после синтеза и проверки находок. Два механизма: **severity-гейт** (жёсткий, для безопасности) и **готовность 0-10** (числовая, для общей картины). Гейт перекрывает балл: один закоммиченный приватный ключ - провал при любом балле.

## Severity

| Severity | Определение | Примеры |
|----------|-------------|---------|
| **Critical** | Прямой компромисс: живой секрет/ключ в дереве или истории; RCE; публичный доступ к приватным данным | приватный ключ в репо; `pickle.loads` на теле запроса; AWS-ключ в истории git |
| **High** | Вероятная эксплуатируемая уязвимость с реальным impact | SQL-инъекция с подтверждённым source→sink; обход authz/IDOR |
| **Medium** | Слабое место без прямого пути эксплуатации или с ограниченным impact | CORS `*` без credentials; слабый хеш; debug-эндпоинт за auth |
| **Low** | Гигиена, hardening | нет паттернов в `.gitignore`; устаревшая, но не уязвимая зависимость |
| **Info** | Наблюдение, рекомендация на будущее | стоит добавить security-заголовки |

Для подготовки к **public**: любой секрет в рабочем дереве **или истории** = Critical автоматически, даже «тестовый/демо». Считай его скомпрометированным.

## Готовность (0-10)

По 0-2 за категорию, как в DotCore-аудите:

| Категория | 0 | 1 | 2 |
|-----------|---|---|---|
| Секреты/ключи (трек A) | есть живой секрет | подозрения/гигиена | чисто, история проверена |
| PII/экспозиция (трек A) | приватные данные/инфра наружу | мелочи | чисто |
| Код: инъекции/exec (трек B) | High/Critical | Medium | не найдено по охвату |
| Код: authz/крипто/дефолты (трек B) | High/Critical | Medium | ок |
| Зависимости/инфра/CI | известная уязвимость High+ | устаревшее/Medium | ок |

Балл - ориентир. Гейт ниже - решает выдачу бейджа.

## Гейт

```
если есть открытые Critical или High  -> FAILED          (бейджа нет)
иначе если есть открытые Medium        -> PASSED WITH WARNINGS
иначе                                  -> PASSED
```

- **FAILED** - бейдж и файл отчёта не выдаются. Если в README уже есть блок от прошлого PASS - **снять его** вместе с `SECURITY-AUDIT.md` (не оставлять устаревшее заявление о пройденном аудите).
- **PASSED WITH WARNINGS** - бейдж выдаётся (жёлтый), в отчёте перечислены Medium; для public рекомендуется устранить и перепроверить.
- **PASSED** - бейдж выдаётся (зелёный).

Охват фиксируется в файле отчёта и в `alt`-тексте бейджа (`утечки`, `код`, `утечки + код`) - PASS по треку A не заявляет аудит кода.

## Формат отчёта (вывести пользователю)

```
Pre-Deploy Audit - {уровень}, охват: {утечки | код | утечки + код}
──────────────────────────────────────────────
Трек A · Секреты/ключи:   {n}  (Crit {x} / High {y})
Трек A · PII/экспозиция:   {n}
Трек A · История git:      {n}            (полный)
Трек B · Инъекции/exec:    {n}
Трек B · Authz/крипто:     {n}
Трек B · Зависимости:      {n}
Инфра/CI:                  {n}            (полный)
──────────────────────────────────────────────
Severity: Crit {} · High {} · Med {} · Low {} · Info {}
Готовность: X/10
Вердикт: PASSED | PASSED WITH WARNINGS | FAILED

Артефакты: блок audit в README + docs/audit/{дата}-{слово}.md + latest.md | не выданы - {причина}
```

Затем таблица находок (severity по убыванию):

| Severity | Категория | Файл:строка | Описание | Рекомендация |
|----------|-----------|-------------|----------|--------------|

Улики в таблице - маскированные. Значения секретов не выводить.

## Материализация на PASS

На PASS этот же отчёт записывается в **`docs/audit/{дата}-{слово}.md`** (снимок прогона: дата + кодовое слово, чтобы несколько прогонов за день не затирали друг друга - видимая история) и копируется в **`docs/audit/latest.md`** (стабильная цель бейджа `security_audit`). Markdown-версия: шапка статус/уровень/охват/модель/дата + сводка + таблица находок. Формат файлов, бейдж и миграция со старого 5-бейджевого формата - в [badge.md](badge.md). В файлы отчётов, как и в вывод, **значения секретов не идут** - только `file:line` и маски.

## Приоритет ремедиации

1. Critical-утечки: ротация ключа → удаление из дерева → история (вручную).
2. Critical/High в коде: исправить путь source→sink.
3. Medium: устранить до публикации, где разумно.
4. Low/Info: бэклог hardening.
