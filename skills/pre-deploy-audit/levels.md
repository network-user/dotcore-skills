# Уровни и треки

Две независимые оси. **Трек** = что проверяем, **уровень** = насколько глубоко.

## Оси

- **Трек A - утечки** ([track-leaks.md](track-leaks.md)): данные, которые нельзя выкатывать. Паттерн-скан, без понимания логики.
- **Трек B - код** ([track-code.md](track-code.md)): уязвимости в логике. Чтение и разбор кода.

Лёгкий способ запомнить: трек A отвечает «не утечёт ли что-то приватное, если репозиторий станет публичным», трек B - «можно ли этот код сломать/проэксплуатировать».

## Матрица

### Поверхностный

Минуты, без подагентов. Цель - поймать очевидные foot-guns до деплоя.

| | Что делает |
|---|---|
| Трек A | Скан **рабочего дерева** (`git ls-files`) на секреты/ключи/токены по паттернам; проверка, что `.env*`, `*.pem`, `*.key`, `credentials*` не в индексе и покрыты `.gitignore`; machine-paths (`C:\Users\…`, `/home/<user>`); закоммиченные артефакты сборки |
| Трек B | Только **явные красные флаги** грепом: `eval(`, `exec(`, `os.system`, `subprocess(..., shell=True)`, `child_process.exec`, `pickle.loads`, hardcoded пароли/ключи в коде, `DEBUG = True`, `verify=False`. Без глубокого чтения логики |
| Подагенты | Нет, линейно |

### Средний (дефолт)

Десятки минут, 2-4 подагента. Цель - готовность к деплою: данные + основные уязвимости в значимых местах.

| | Что делает |
|---|---|
| Трек A | Всё из поверхностного + экспозиция: открытые порты/привязки `0.0.0.0`, CORS `*`, отключённая проверка TLS, секреты как значения по умолчанию в конфигах, `.env.example` vs реально требуемые переменные |
| Трек B | Ревью **критичных и изменённых** областей (точки входа, auth, обработка ввода, работа с БД/файлами/сетью) на инъекции, десериализацию, path traversal, authz; аудит **зависимостей** (известные уязвимости, устаревшие, typosquat, postinstall-скрипты) |
| Подагенты | 2-4 параллельно по измерениям |

### Полный

Часы, веер подагентов + проверочный проход. Цель - исчерпывающе; режим для смены видимости на public и для серьёзного релиза.

| | Что делает |
|---|---|
| Трек A | Всё из среднего + **история git** (секреты, удалённые из дерева, но оставшиеся в истории), PII вглубь (email, имена, внутренние хосты/IP, клиентские данные), supply chain (целостность lockfile, источники пакетов) |
| Трек B | **Вся кодовая база** по модулям, семантический разбор data-flow (source → sink) по всем категориям [track-code.md](track-code.md); инфра (Dockerfile, права, секреты в слоях/ARG) и CI (`pull_request_target`, эхо секретов, workflow-инъекции) |
| Подагенты | Веер 6-10 по измерениям + **adversarial-проверка** каждой находки Critical/High |

## Фокус: один трек или оба

Уровень - глубина, фокус - выбор трека(ов). Они независимы:

- **Только утечки на максимум** (типичный «перед публикацией»): трек A на полном уровне (**с историей git**), трек B можно опустить или ограничить явными флагами. Не нужно гнать многочасовой код-аудит, чтобы убедиться, что нет утёкших ключей.
- **Только аудит кода**: трек B вглубь (обычно полный), трек A как гигиена (быстрый скан дерева).
- **Оба** (дефолт перед деплоем/релизом): A и B на выбранном уровне.

В отчёте и на бейджах фиксируй охват: `утечки`, `код` или `утечки + код` - чтобы бейджи не заявляли больше, чем проверено.

## Выбор по запросу

| Запрос | Трек | Уровень |
|--------|------|---------|
| «нет ли секретов / утечек» | A | поверхностный / средний |
| «делаю репо публичным» | A + история git | полный (A) |
| «быстро перед деплоем» | A + B (флаги) | поверхностный |
| «перед релизом» | A + B | средний |
| «полный аудит / весь код» | A + B | полный |

Неоднозначно между «утечки» и «весь код» - уточни одним вопросом. Не подменяй дешёвый запрос дорогим и наоборот.
