# Бейдж аудита и файлы отчётов

Выдаётся **только на PASSED** (или PASSED WITH WARNINGS). На FAILED ничего не выдаётся; существующий блок бейджа от прошлого PASS - снять (история в `docs/audit/` остаётся, см. ниже).

Артефакт на PASS - **два уровня**:

1. **Файлы отчётов** в `docs/audit/`:
   - `docs/audit/{DATE}-{SLUG}.md` - снимок прогона (накапливается, видимая история аудитов). Имя = дата + кодовое слово прогона, см. [Именование снимков](#именование-снимков).
   - `docs/audit/latest.md` - копия последнего снимка, **стабильная цель бейджа** `security_audit` (её `href` никогда не меняется).
2. **Блок в README** - компактный: **кликабельный** flat-бейдж `security_audit` (ведёт на `docs/audit/latest.md` - всегда последний прогон) плюс **кликабельный** серый бейдж `date` (ведёт на снимок этого прогона `docs/audit/{DATE}-{SLUG}.md`). Без картинки, в общем стиле header-бейджей README.

Принцип: шапка README несёт минимум (пройдено + когда), детали - по клику в отчёте. Уровень, охват и модель в шапку не выносятся - они в файлах отчётов и в `alt`-тексте бейджа.

## Именование снимков

Имя снимка: **`docs/audit/{DATE}-{SLUG}.md`**, где

- `{DATE}` - ISO-дата прогона, `YYYY-MM-DD` (напр. `2026-07-01`).
- `{SLUG}` - короткое **кодовое слово** прогона в kebab-case, обычно два слова прилагательное-существительное (`swift-harbor`, `amber-falcon`, `iron-gate`). Агент выбирает его при прогоне.

Зачем слово, а не только дата: **аудит могут запускать несколько раз за день**. Одна дата в имени затирала бы предыдущий снимок; дата + кодовое слово даёт уникальность в пределах дня и читаемое имя (лучше запоминается, чем время `HHMM`). Разные прогоны за день → разные слова → снимки не конфликтуют.

Fallback, если кодовое слово не задаётся: числовой суффикс по порядку в течение дня - `2026-07-01-01.md`, `2026-07-01-02.md`.

`latest.md` всегда содержит копию последнего снимка - на него ссылается бейдж `security_audit`. Бейдж `date` ссылается на конкретный снимок `{DATE}-{SLUG}.md` этого прогона.

Не используй `SECURITY.md` - оно зарезервировано под policy раскрытия уязвимостей.

## Формат файла отчёта

Содержимое снимка - markdown-версия отчёта из [report.md](report.md):

```markdown
# Security Audit · Swift Harbor · 2026-07-01

| Поле | Значение |
|------|----------|
| Статус | PASSED |
| Прогон | swift-harbor |
| Уровень | full |
| Охват | leaks + code |
| Модель | Claude Opus 4.8 |
| Дата | 2026-07-01 |

## Сводка

Трек A · Секреты/ключи:   0  (Crit 0 / High 0)
Трек A · PII/экспозиция:   0
Трек A · История git:      0
Трек B · Инъекции/exec:    0
Трек B · Authz/крипто:     0
Трек B · Зависимости:      0
Инфра/CI:                  0

Severity: Crit 0 · High 0 · Med 0 · Low 1 · Info 2
Готовность: 10/10
Вердикт: PASSED

## Находки

| Severity | Категория | Файл:строка | Описание | Рекомендация |
|----------|-----------|-------------|----------|--------------|
| Low | … | … | … | … |
```

`latest.md` идентичен последнему снимку плюс строка-пометка в начале, что это последний прогон и ссылка на датированный снимок:

```markdown
> Последний прогон: swift-harbor · 2026-07-01. Снимок: [2026-07-01-swift-harbor.md](2026-07-01-swift-harbor.md) · история: [docs/audit/](.)
```

Жёстко: **значения секретов в отчёт не идут** - только `file:line` + категория + маскированная улика (как в [report.md](report.md)).

## Бейджи в README

`style=flat`, оба в одном `<p>` (GitHub-first - иначе разъезжаются).

| Бейдж | Значение | Цвет | Кликабельный |
|-------|----------|------|--------------|
| `security_audit` | `passed` | `3fb950` зелёный | да → `docs/audit/latest.md` |
| | `passed_with_warnings` | `dbab09` жёлтый | да → `docs/audit/latest.md` |
| `date` | дата аудита | `555` серый | да → `docs/audit/{DATE}-{SLUG}.md` (снимок этого прогона) |

- `security_audit` обёрнут в `<a href="docs/audit/latest.md">`. Цвет кодирует статус: passed зелёный, passed_with_warnings жёлтый.
- Уровень и охват - в `alt`-тексте бейджа (`security audit passed - full, leaks + code`), не отдельными бейджами. Модель - только в файле отчёта.
- `date` - дата дня аудита, серая, обёрнута в `<a href="docs/audit/{DATE}-{SLUG}.md">` (снимок этого прогона). Делёж ролей: `security_audit` → `latest.md` (стабильная ссылка на последний прогон), `date` → датированный снимок (исторический якорь).
- Важно: в пути `href` дата **без** экранирования (`docs/audit/2026-07-01-swift-harbor.md`), а в значении бейджа `date` дефисы экранированы (`2026--07--01`). Не перепутай.

FAILED - блок бейджа в README не выдаётся; если есть от прошлого PASS - удалить (снять блок вместе с маркерами). Снимки в `docs/audit/` не стираем - это история; но `latest.md` от провалившегося прогона не пишем.

Экранирование значений в URL бейджа: `-` → `--`, пробел → `_`, `+` → `%2B`. В `alt`-тексте (обычный HTML-текст) и в `href` экранирование не нужно.

## Скелет

```markdown
<!-- audit:start -->
<p>
  <a href="docs/audit/latest.md"><img src="https://img.shields.io/badge/security_audit-{STATUS}-{STATUS_COLOR}?style=flat" alt="security audit {STATUS_TXT} - {LEVEL_EN}, {SCOPE_TXT}" /></a>
  <a href="docs/audit/{DATE_ISO}-{SLUG}.md"><img src="https://img.shields.io/badge/date-{DATE}-555?style=flat" alt="audit date" /></a>
</p>
<!-- audit:end -->
```

`{STATUS}` = `passed` / `passed_with_warnings`; `{STATUS_COLOR}` = `3fb950` / `dbab09`; `{LEVEL_EN}` = `surface` / `medium` / `full`; `{SCOPE_TXT}` = `leaks` / `code` / `leaks + code`; `{DATE}` = ISO-дата с экранированием `-`→`--` (в значении бейджа); `{DATE_ISO}` = та же дата **без** экранирования; `{SLUG}` = кодовое слово прогона (в пути `href`, напр. `2026-07-01-swift-harbor`).

Пример (PASS, полный, оба трека):

```markdown
<!-- audit:start -->
<p>
  <a href="docs/audit/latest.md"><img src="https://img.shields.io/badge/security_audit-passed-3fb950?style=flat" alt="security audit passed - full, leaks + code" /></a>
  <a href="docs/audit/2026-07-01-swift-harbor.md"><img src="https://img.shields.io/badge/date-2026--07--01-555?style=flat" alt="audit date" /></a>
</p>
<!-- audit:end -->
```

Путь в `href` относителен расположения README. README в корне → `docs/audit/...`. README в подкаталоге → скорректируй относительный путь.

## Вставка в README

Блок **additive**, в маркерах - чтобы перезапуск аудита обновлял его на месте, не дублируя (как `loc:start/end` в generate-readme). Размести после обложки/шапки.

- Маркеры `<!-- audit:start -->` / `<!-- audit:end -->` обязательны и принадлежат этому скиллу. Не удалять.
- Перезапуск аудита: перепиши содержимое между маркерами (статус, дата; `href` бейджа `security_audit` остаётся `docs/audit/latest.md`, `href` бейджа `date` укажи на новый снимок `docs/audit/{DATE}-{SLUG}.md`), добавь снимок `docs/audit/{DATE}-{SLUG}.md`, обнови `latest.md`. Не плоди второй блок.
- Совместимость с `generate-readme`: при полной перегенерации README тот скилл **переносит блок дословно** (не выдумывает и не правит) и не трогает `docs/audit/` - см. `skills/generate-readme/stack-badges.md`. Поэтому бейдж и отчёты переживают регенерацию README.

## Миграция со старого формата

Ранее блок был рядом из пяти цветных бейджей без ссылки: `security_audit`, `level`, `scope`, `model`, `date`, а файла отчёта не было. Ещё раньше снимки именовались просто `{DATE}.md` (без кодового слова) - это ломалось при повторном прогоне за день. Новый формат - два кликабельных бейджа: `security_audit` (→ `docs/audit/latest.md`) + `date` (→ `docs/audit/{DATE}-{SLUG}.md`), детали в `docs/audit/`.

При прогоне аудита, если между маркерами найден **старый** блок (есть бейджи `level`/`scope`/`model`, либо `security_audit` не обёрнут в `<a>`, либо снимок назван без кодового слова):

1. Считай уровень/охват/модель/дату из старых бейджей.
2. Перепиши блок в новый формат: кликабельный `security_audit` (`href` → `docs/audit/latest.md`) + кликабельный `date` (`href` → `docs/audit/{DATE}-{SLUG}.md`), уровень и охват перенеси в `alt`-текст.
3. Создай снимок `docs/audit/{DATE}-{SLUG}.md` (по дате из старого бейджа + кодовое слово) и `docs/audit/latest.md`, перенеся уровень, охват, модель, дату; находки - из текущего прогона, либо пометь, что детальные находки от прошлого прогона не сохранялись, статус clean.

Миграция формата - не новый вердикт: если повторный аудит не гонялся, перенеси старое заявление как есть, дату не меняй. Полноценный новый прогон добавляет новый снимок по текущей дате+слову и обновляет `latest.md`, бейдж, дату.

## Целостность (важно)

Бейдж - заявление «аудит уровня X пройден на дату Y», файлы в `docs/audit/` - его раскрытие. Они не должны врать:

- **Только на PASS.** FAILED - бейдж не выдавать; `latest.md` от провала не писать; если блок уже есть - удалить.
- **Дата - дня аудита.** Не переноси старую дату на новый прогон (кроме чистой миграции формата без переаудита).
- **Охват честный.** Трек A → `alt` и отчёт пишут `leaks`; не заявляй `leaks + code`, если код не аудировали.
- **Отчёт без секретов.** В файлы `docs/audit/` идут `file:line` и маски, не значения.
- Существенные изменения кода после выдачи обесценивают заявление - перезапусти аудит, добавь снимок, обнови `latest.md`, дату и бейдж.
