# Бейдж аудита и файлы отчётов

Выдаётся **только на PASSED** (или PASSED WITH WARNINGS). На FAILED ничего не выдаётся; существующий блок бейджа от прошлого PASS - снять (история в `docs/audit/` остаётся, см. ниже).

Артефакт на PASS - **два уровня**:

1. **Файлы отчётов** в `docs/audit/`:
   - `docs/audit/{DATE}.md` - снимок прогона (накапливается, видимая история аудитов).
   - `docs/audit/latest.md` - копия последнего снимка, **стабильная цель бейджа** (её `href` никогда не меняется).
2. **Блок в README** - компактный: один **кликабельный** flat-бейдж `security_audit` (ведёт на `docs/audit/latest.md`) плюс серый бейдж `date`. Без картинки, в общем стиле header-бейджей README.

Принцип: шапка README несёт минимум (пройдено + когда), детали - по клику в отчёте. Уровень, охват и модель в шапку не выносятся - они в файлах отчётов и в `alt`-тексте бейджа.

Почему два файла, а не один: бейдж заявляет текущее состояние и должен вести на **один неизменный путь** (`latest.md`); снимки по дате дают историю прогонов файлами, не полагаясь на `git log`. Снимки изолированы в `docs/audit/`, корень репозитория чист.

## Файлы отчётов `docs/audit/`

Каталог `docs/audit/` в корне репозитория. Имя снимка - ISO-дата прогона: `docs/audit/2026-06-28.md`. Повторный прогон в ту же дату перезаписывает снимок этой даты. `docs/audit/latest.md` всегда содержит копию последнего снимка - на него ссылается бейдж.

Не используй `SECURITY.md` - оно зарезервировано под policy раскрытия уязвимостей.

Содержимое снимка - markdown-версия отчёта из [report.md](report.md):

```markdown
# Security Audit - 2026-06-28

| Поле | Значение |
|------|----------|
| Статус | PASSED |
| Уровень | full |
| Охват | leaks + code |
| Модель | Claude Opus 4.8 |
| Дата | 2026-06-28 |

## Сводка

Трек A · Секреты/ключи:   0  (Crit 0 / High 0)
Трек A · PII/экспозиция:   0
Трек A · История git:      0
Трек B · Инъекции/exec:    0
Трек B · Authz/крипто:     0
Трек B · Зависимости:      0
Инфра/CI:                  0

Severity: Crit 0 · High 0 · Med 0 · Low 1 · Info 2
Готовность: 10/10
Вердикт: PASSED

## Находки

| Severity | Категория | Файл:строка | Описание | Рекомендация |
|----------|-----------|-------------|----------|--------------|
| Low | … | … | … | … |
```

`latest.md` идентичен последнему снимку плюс строка-пометка в начале, что это последний прогон и ссылка на датированный снимок:

```markdown
> Последний прогон. Снимок: [2026-06-28.md](2026-06-28.md) · история: [docs/audit/](.)
```

Жёстко: **значения секретов в отчёт не идут** - только `file:line` + категория + маскированная улика (как в [report.md](report.md)).

## Бейджи в README

`style=flat`, оба в одном `<p>` (GitHub-first - иначе разъезжаются).

| Бейдж | Значение | Цвет | Кликабельный |
|-------|----------|------|--------------|
| `security_audit` | `passed` | `3fb950` зелёный | да → `docs/audit/latest.md` |
| | `passed_with_warnings` | `dbab09` жёлтый | да → `docs/audit/latest.md` |
| `date` | дата аудита | `555` серый | нет |

- `security_audit` обёрнут в `<a href="docs/audit/latest.md">`. Цвет кодирует статус: passed зелёный, passed_with_warnings жёлтый.
- Уровень и охват - в `alt`-тексте бейджа (`security audit passed - full, leaks + code`), не отдельными бейджами. Модель - только в файле отчёта.
- `date` - дата дня аудита, серая, не ссылка. Без неё `passed` бессмыслен (см. Целостность).

FAILED - блок бейджа в README не выдаётся; если есть от прошлого PASS - удалить (снять блок вместе с маркерами). Снимки в `docs/audit/` не стираем - это история; но `latest.md` от провалившегося прогона не пишем.

Экранирование значений в URL бейджа: `-` → `--`, пробел → `_`, `+` → `%2B`. В `alt`-тексте (обычный HTML-текст) и в `href` экранирование не нужно.

## Скелет

```markdown
<!-- audit:start -->
<p>
  <a href="docs/audit/latest.md"><img src="https://img.shields.io/badge/security_audit-{STATUS}-{STATUS_COLOR}?style=flat" alt="security audit {STATUS_TXT} - {LEVEL_EN}, {SCOPE_TXT}" /></a>
  <img src="https://img.shields.io/badge/date-{DATE}-555?style=flat" alt="audit date" />
</p>
<!-- audit:end -->
```

`{STATUS}` = `passed` / `passed_with_warnings`; `{STATUS_COLOR}` = `3fb950` / `dbab09`; `{LEVEL_EN}` = `surface` / `medium` / `full`; `{SCOPE_TXT}` = `leaks` / `code` / `leaks + code`; `{DATE}` = ISO-дата с экранированием `-`→`--`.

Пример (PASS, полный, оба трека):

```markdown
<!-- audit:start -->
<p>
  <a href="docs/audit/latest.md"><img src="https://img.shields.io/badge/security_audit-passed-3fb950?style=flat" alt="security audit passed - full, leaks + code" /></a>
  <img src="https://img.shields.io/badge/date-2026--06--28-555?style=flat" alt="audit date" />
</p>
<!-- audit:end -->
```

Путь в `href` относителен расположения README. README в корне → `docs/audit/latest.md`. README в подкаталоге → скорректируй относительный путь.

## Вставка в README

Блок **additive**, в маркерах - чтобы перезапуск аудита обновлял его на месте, не дублируя (как `loc:start/end` в generate-readme). Размести после обложки/шапки.

- Маркеры `<!-- audit:start -->` / `<!-- audit:end -->` обязательны и принадлежат этому скиллу. Не удалять.
- Перезапуск аудита: перепиши содержимое между маркерами (статус, дата, `href` остаётся `docs/audit/latest.md`), добавь новый снимок `docs/audit/{DATE}.md`, обнови `latest.md`. Не плоди второй блок.
- Совместимость с `generate-readme`: при полной перегенерации README тот скилл **переносит блок дословно** (не выдумывает и не правит) и не трогает `docs/audit/` - см. `skills/generate-readme/stack-badges.md`. Поэтому бейдж и отчёты переживают регенерацию README.

## Миграция со старого формата

Ранее блок был рядом из пяти цветных бейджей без ссылки: `security_audit`, `level`, `scope`, `model`, `date`, а файла отчёта не было. Новый формат - один кликабельный `security_audit` + `date`, детали в `docs/audit/`.

При прогоне аудита, если между маркерами найден **старый** блок (есть бейджи `level`/`scope`/`model`, либо `security_audit` не обёрнут в `<a>`):

1. Считай уровень/охват/модель/дату из старых бейджей.
2. Перепиши блок в новый формат (кликабельный `security_audit` + `date`), уровень и охват перенеси в `alt`-текст, `href` → `docs/audit/latest.md`.
3. Создай `docs/audit/{DATE}.md` (по дате из старого бейджа) и `docs/audit/latest.md`, перенеся уровень, охват, модель, дату; находки - из текущего прогона, либо пометь, что детальные находки от прошлого прогона не сохранялись, статус clean.

Миграция формата - не новый вердикт: если повторный аудит не гонялся, перенеси старое заявление как есть, дату не меняй. Полноценный новый прогон добавляет новый снимок по текущей дате и обновляет `latest.md`, бейдж, дату.

## Целостность (важно)

Бейдж - заявление «аудит уровня X пройден на дату Y», файлы в `docs/audit/` - его раскрытие. Они не должны врать:

- **Только на PASS.** FAILED - бейдж не выдавать; `latest.md` от провала не писать; если блок уже есть - удалить.
- **Дата - дня аудита.** Не переноси старую дату на новый прогон (кроме чистой миграции формата без переаудита).
- **Охват честный.** Трек A → `alt` и отчёт пишут `leaks`; не заявляй `leaks + code`, если код не аудировали.
- **Отчёт без секретов.** В файлы `docs/audit/` идут `file:line` и маски, не значения.
- Существенные изменения кода после выдачи обесценивают заявление - перезапусти аудит, добавь снимок, обнови `latest.md`, дату и бейдж.
